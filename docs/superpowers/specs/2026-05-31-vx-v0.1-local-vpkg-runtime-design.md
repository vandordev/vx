# VX v0.1 Local VPKG Runtime Design

## Goal

Define the `vx v0.1` public CLI around a small, deterministic, project-local
runtime for `vxt` and `vpkg`, without introducing remote registry behavior,
package installation flows, or task execution yet.

The goal is to make `vx` useful as a real public CLI now while preserving the
future product boundary:

- `vxt` remains the typed generation engine
- `vpkg` remains the package contract
- `vx` owns source resolution, project detection, preview, input gathering, and
  apply behavior

## Scope

This design covers:

- replacing the current root default behavior with command-centric CLI behavior
- local project discovery through a parent `vpkg/` lookup
- local package discovery from `vpkg/<namespace>/<package>/vpkg.yaml`
- direct `.vxt` source support inside the detected project root
- manifest shape and validation rules for local `vpkg.yaml`
- source resolution rules for full references, shorthand, and direct file paths
- `vx view` behavior for package, export, and direct `.vxt` targets
- `vx gen` behavior for preview and apply
- interactive input completion for missing fields only
- initial JSON output support

## Non-Goals

This design does not include:

- remote registry support
- `vx add`
- `vx exec`
- native dependency installation
- trust policy persistence beyond the minimal runtime shape needed later
- raw shell task execution
- output cache or package index cache
- README-driven package metadata
- TUI-first command output for `view` or `gen`

## Recommended Approach

Implement `vx v0.1` as a dual-source local runtime:

- package/export sources resolved from project-local `vpkg/`
- direct `.vxt` file sources resolved from paths under the same project root

This keeps `vx` native to both `vpkg` and `vxt` while staying fully local and
preview-first.

Rationale:

- package-only support would make `vx` feel too tightly coupled to `vpkg`
- direct-`.vxt`-only support would weaken the package contract too early
- local-only scope proves the core command model before registry and execution
  complexity arrive

## Command Surface

`vx v0.1` should expose the following public command surface:

- `vx`
- `vx view <target>`
- `vx gen <target>`
- `vx generate <target>` as an alias for `vx gen`

`vx` without arguments should no longer open the inherited directory browser.
Instead, it should print a short overview with examples for `view` and `gen`.

`vx add` and `vx exec` are intentionally delayed.

Rationale:

- `gen` is short, ergonomic, and suitable as the primary public verb
- `generate` remains useful as a compatibility alias and for documentation
- delaying `add` and `exec` prevents semantic overreach before package install
  and task execution models are mature

## Project Model

`vx` should be project-local in `v0.1`.

### Project root detection

When any `view` or `gen` command runs, `vx` should walk upward from the current
working directory until it finds the nearest parent directory that contains
`vpkg/`.

That directory becomes the detected project root.

If no such directory exists, the command fails.

### Output root

The detected project root is also the default output root for `vx gen`,
including direct `.vxt` sources.

The output root should not depend on the user's current working directory once
the project root has been detected.

### Direct `.vxt` source boundary

Direct `.vxt` files are valid only when:

- the file extension is `.vxt`
- the file is located under the detected project root

Direct `.vxt` sources outside the detected project root must fail.

## Local Package Layout

`vx` should discover packages only from the detected project root:

```text
<project-root>/
  vpkg/
    <namespace>/
      <package>/
        vpkg.yaml
        templates/
        files/
        docs/
```

Only directories containing `vpkg.yaml` are treated as valid packages.

Folders that merely contain `.vxt` files are not packages unless they also
contain a valid manifest.

Direct file execution remains available through explicit `.vxt` paths and is a
separate source class from package discovery.

## Manifest Model

The package manifest for `v0.1` is `vpkg.yaml`.

### Required package fields

- `apiVersion`
- `name`
- `version`
- `kind`
- `exports`

### Required identity rules

- `name` must be namespace-qualified
- `name` must match the package folder path under `vpkg/`

Example:

```text
vpkg/vandor/go-backend-core/vpkg.yaml
```

must declare:

```yaml
name: vandor/go-backend-core
```

Mismatch between manifest name and folder path must fail validation.

### Required package `kind`

Package `kind` is required in `v0.1`.

Supported values:

- `template-pack`
- `preset`
- `tooling`

Even though runtime behavior is limited initially, package kind should remain
explicit so that package intent is self-describing and future-safe.

### Required `version`

`version` is required even for local-only packages.

Reason:

- keeps local and future registry contracts aligned
- avoids future migration from optional to required identity metadata

## Export Model

Each package must declare one or more exports.

Export `kind` is required from the start.

Supported values:

- `template`
- `preset`
- `task`

`default` is the recommended main export name and receives implicit resolution
when a package reference omits `:export`.

### `template` export rules

For `kind: template`:

- `templates` is required
- `templates` must contain at least one item
- every `templates[].path` must be relative to the package root
- every `templates[].path` must remain inside the package root
- every `templates[].path` must point to an existing `.vxt` file

If one template export references multiple `.vxt` files, all templates must
share the same input contract.

`vx` should fail package loading if the input contracts differ.

This keeps one export equal to one input shape.

### `preset` export rules

For `kind: preset`:

- manifest parsing and validation are supported in `v0.1`
- `vx view` should display declared template/file/directory shape
- runtime application is out of scope for `v0.1`

`preset` exists in the schema now so package authors can write forward-looking
manifests without waiting for `vx add`.

### `task` export rules

For `kind: task`:

- manifest parsing and validation are supported in `v0.1`
- `vx view` should display declared task steps
- runtime execution is out of scope for `v0.1`

Task steps should be limited to built-in step types only.

Allowed early step types:

- `check-file`
- `check-command`
- `check-env`
- `format`
- `native-install`

Raw shell-like steps should not be supported in `v0.1`.

## Source Resolution

`vx` must support two source classes in `v0.1`:

- local package/export references
- direct `.vxt` paths

### Supported package reference forms

- `namespace/package`
- `namespace/package:export`
- shorthand package if unique
- shorthand export if unique

Examples:

- `vandor/go-backend-core`
- `vandor/go-backend-core:usecase`
- `go-backend-core`
- `usecase`

### Direct file reference form

- `./templates/usecase.vxt`

### Resolution behavior

Resolution order:

1. if the target is a `.vxt` path, treat it as a direct template source
2. otherwise try full package reference resolution
3. otherwise try shorthand package resolution
4. otherwise try shorthand export resolution

If shorthand resolution produces more than one candidate, `vx` must fail with a
clear candidate list.

No automatic tie-breaking should be introduced in `v0.1`.

## `vx view`

`vx view` is the primary non-destructive inspection command.

### Output mode

Default output should be structured text.

`--json` should be supported in `v0.1`.

TUI output is intentionally deferred.

### Package target behavior

When the target is a package without `:export`, `vx view` should show:

- package identity
- package version
- package kind
- all exports with their export kind

It should not auto-resolve to `default` in this package-summary mode.

### Export target behavior

When the target resolves to a specific export:

- for `template`, show export metadata, referenced template files, required
  inputs from `.vxt`, and planned output only when `--plan` is requested and
  input is sufficient
- for `preset`, show export metadata plus declared `templates`, `files`,
  `directories`, and declared actions as metadata only
- for `task`, show export metadata plus ordered step details and clearly mark
  the runtime as view-only in `v0.1`

### Planning behavior

`vx view` should be staged:

- without `--plan`, show metadata and input contract only
- with `--plan`, compile, validate, and plan via `vxt`
- if required input is missing, fail with a focused missing-input error

### README handling

`vx view` should ignore package `README.md` in `v0.1`.

Source of truth should remain:

- `vpkg.yaml` for package/export identity and shape
- `.vxt` for input contract

## `vx gen`

`vx gen` is the primary generation command in `v0.1`.

`vx generate` should behave as an alias to the same command.

### Supported executable sources

Only these source targets are executable in `v0.1`:

- direct `.vxt` files
- `template` exports

`preset` and `task` exports are not executable in `v0.1`.

### Default behavior

`vx gen` must be preview-first.

Default flow:

1. resolve source
2. detect project root
3. gather input
4. compile and validate via `vxt`
5. build plan
6. print preview

Files are written only when `--apply` is passed explicitly.

There should be no interactive confirmation prompt as the default write gate.

### Output root behavior

All planned and applied writes target the detected project root, not the
current working directory.

### Conflict behavior

If a planned output file already exists, the default behavior is to fail.

This applies to `--apply`.

Preview output should surface the conflict list so the user can see the issue
before attempting a write.

### Preview output

Default preview output should include:

- resolved source
- resolved export when applicable
- project root
- planned directories
- planned files
- conflicting existing files
- diagnostics

Rendered file content should not be shown by default.

### JSON mode

`--json` should be supported for preview and apply result reporting.

The JSON shape should be a CLI contract summary, not a dump of internal `vxt`
structures.

## Input Gathering

Input for `view --plan` and `gen` should be collected in this order:

1. `--values <file>`
2. `--set key=value`
3. interactive prompt for missing fields only

### `--values`

`--values` should support:

- YAML
- JSON

### `--set`

`--set` should support:

- scalar values
- dotted paths for simple nested assignment

Example:

```bash
vx gen usecase --set context=booking --set name=create_booking
```

### Prompting rules

Prompting should happen only for missing fields.

If all required input is already present, no prompt should appear.

If a value is present but invalid, `v0.1` should fail with a clear validation
error instead of prompting for correction.

If `--non-interactive` is active, missing required input must fail instead of
prompting.

### Input source of truth

Input contract and validation should come from the `.vxt` templates themselves,
not from duplicated schema declarations in `vpkg.yaml`.

This avoids manifest-template drift and keeps `vxt` as the true typed input
boundary.

## Root Command UX

`vx` without arguments should print a short overview with example usage.

Recommended content:

- what `vx view` does
- what `vx gen` does
- examples for package/export and direct `.vxt` usage
- hint that the current project must define `vpkg/`

The inherited interactive directory browser should not remain the default root
experience for the public CLI.

## Error Position

The `v0.1` CLI should prefer explicit failure over hidden fallback behavior.

Examples:

- no project root found -> fail
- ambiguous shorthand -> fail with candidates
- direct `.vxt` outside project root -> fail
- missing `default` export when package execution implies it -> fail
- existing file conflict on apply -> fail

This keeps the CLI scriptable, testable, and predictable.

## Testing Strategy

Verification for implementation should cover:

- project root detection by nearest parent `vpkg/`
- package discovery from local filesystem without cache
- manifest validation
- path-based direct `.vxt` validation
- shorthand resolution and ambiguity failure
- package summary view
- export view for `template`, `preset`, and `task`
- preview-only `gen`
- `--apply` write behavior
- file conflict failure
- `--values` plus `--set` precedence
- interactive prompting only for missing inputs
- JSON output shape

## Success Criteria

`vx v0.1` is considered successful when:

- it behaves as a project-local public CLI instead of a generic directory tool
- it can discover local `vpkg` packages from the nearest project root
- it can inspect package/export metadata through `vx view`
- it can preview and apply `template` exports and direct `.vxt` sources through
  `vx gen`
- it keeps `vxt` as the engine boundary and does not duplicate template input
  contracts in the manifest
- it remains deterministic, preview-first, and safe-by-default
